1. 产品概述
产品是什么: TasteBridge，以地图为核心的探店记忆共享平台。用户在餐饮店点位记录个人“喜爱值”、情绪标签、图片和文字评价，形成私人情绪地图；支持邮箱注册登录，与好友共建共享。

目标用户: 活跃于城市、喜欢探索美食、习惯用Web端记录分享的年轻用户。

解决什么核心问题: 传统点评平台评分趋同、缺乏个人情感连接；美食记忆零散，无法与地理位置和亲密社交深度绑定；探店记录缺乏有效的分类和检索方式。

为什么现在应该存在: 高德地图API成熟，通义千问API可低成本实现标签推荐和语义搜索；Web端PWA可接近原生体验。

2. 目标
主要目标: 用户能在1.5分钟内完成一次探店记录（搜索店铺 → 自动填充信息 → 添加个人评价 + AI推荐标签）。

次要目标: 用户能与好友共同记录一起去过的店铺（邀请协助填写），并在个人中心查看共同回忆和月度总结。

成功指标 / 可衡量的结果:

首次打开后完成记录的用户占比 ≥ 50%

用户对AI推荐标签的采纳率 ≥ 40%

语义搜索找到目标点位的准确率 ≥ 70%

月度总结查看率 ≥ 已激活用户的30%

发起邀请协助后，好友完成率 ≥ 20%

3. 目标用户
主要用户: 城市美食探索者、探店爱好者、习惯用Web端管理生活记录的用户。

关键用户特征: 喜欢分类整理；对“总结回顾”类内容有共鸣；愿意在小圈子内分享真实感受；有固定的探店搭子。

核心使用场景:

饭后一起记录：“这家川菜不错，我发起邀请，让朋友也来打分”

搜索回忆：“找一家适合约会的日料店”——用语义搜索

月度回顾：“看看这个月吃了什么，AI给我推荐一首歌”

共同回忆：“看看我和朋友一起去过的那些店”

4. 问题陈述
当前痛点:

手动打标签繁琐，用户懒得分类

传统搜索只能按店名/菜系，无法用“适合约会”“辣度适中”等模糊语义查找

探店记录散落各处，缺少仪式感的回顾（月度总结）

和朋友一起探店后，想收集双方评价但缺少协作工具

一起去过的店分散在各自记录中，没有聚合展示

现有替代方案:

大众点评/小红书：搜索靠关键词匹配，无个性化语义理解

Notion/Excel：无AI辅助分类，无地理可视化，无协作

苹果/谷歌地图标记：冷冰冰，无情感连接和回顾功能

为什么现有方案不够好: 缺乏“AI辅助分类 + 语义搜索 + 个性化回忆 + 协作共建”的一体化地理记忆工具。

5. MVP 范围
范围内
基于高德地图的点位浏览（Web端，React + TypeScript）

用户系统：邮箱注册/登录（Supabase Auth）

个人中心：

编辑个人资料（头像、昵称）

头像：默认生成（昵称首字母/默认图标），MVP不支持自定义上传

昵称：可编辑，默认邮箱前缀

打卡记录列表（只读展示，不包含点击跳转功能）

展示所有自己的点位，按时间倒序

每个记录显示：情绪emoji、喜爱值、图片缩略图（最多1张）、店铺名称、文字评价摘要（可选）

共同回忆模块入口（选择特定好友，展示和其一起去过的店铺）

月度总结模块入口（月度总结查看页，无自动通知）

点位添加（仅通过搜索）：

店铺名称搜索框（必填）+ 城市搜索框（选填，缩小范围）

调用高德地图POI搜索API，返回店铺列表

选择店铺后，自动填充：店名、地址、电话、营业时间、人均价格、评分（高德提供）

点位详情页包含：

基础信息：店名、地址、电话、营业时间、人均、评分（只读，来自高德）

个人评价：喜爱值（1-10分）、情绪标签（4个emoji：🥰惊喜 / 😌舒服 / 🤔一般 / 😤踩雷）、图片上传（最多3张）、文字评价（可选）

AI推荐标签：3-5个标签（如#辣味十足 #环境不错），用户可点击高亮/取消选中，支持自定义添加

AI标签推荐：基于店名+菜系（如有）+用户文字评价，调用千问API生成，目的是便于后续语义化搜索分类

添加/编辑/删除自己的点位

个人地图视图：按喜爱值颜色区分

语义化搜索：自然语言输入（如“适合约会的安静餐厅”），调用千问API理解意图，匹配点位的标签+情绪+文字评价

店铺名称搜索：传统关键词搜索（按店名/地址），与语义搜索分开两个入口

好友系统：通过邮箱搜索/添加好友，双向确认（Supabase表存储好友关系）

好友地图视图：同时显示自己和好友的点位

邀请协助功能（一起去探店时使用）：

用户A在添加/编辑点位时，选择“邀请好友共同评价” → 从好友列表选择好友B

系统生成邀请记录，好友B收到通知

好友B点击通知 → 进入点位详情页 → 填写自己的喜爱值、情绪标签、文字评价、图片

提交后，该点位在双方地图上都显示，且标记为“共同去过”

店铺详情卡片中增加“共同去过”Tab：只显示标记为“共同去过”的评价对（双方都填写了邀请协助的评价）

店铺详情卡片（基于高德POI ID聚合）：

同一店铺被多个用户记录时，地图上只显示一个聚合Marker

点击Marker后弹出店铺详情卡片，包含：

店铺基础信息（高德数据）

“共同去过”Tab：只显示邀请协助完成的评价对

若无共同回忆，不显示任何内容（Tab区域为空或隐藏Tab）

共同回忆模块（放在个人中心）：

入口：个人中心 → 共同回忆

先选择特定好友（好友列表）

选择好友后，展示与该好友一起去过的店铺列表（基于邀请协助完成的店铺）

每个店铺显示：店名、地址、双方评价摘要（喜爱值、情绪emoji、文字评价片段）

生成专属总结：共同探店数量、口味相似度（标签重叠率）、情绪重合度

AI推荐一首音乐（可“换一首”）

月度总结模块（放在个人中心）：

入口：个人中心 → 月度总结（无自动通知，用户主动点击查看）

按月展示（支持切换月份）

统计内容：点位数量（当月新增）、平均喜爱值、最常标签Top3、情绪标签分布（🥰/😌/🤔/😤各多少次）

AI推荐一首音乐（可“换一首”）

展示当月打卡记录列表（快速回顾，仅展示店名、日期、喜爱值、情绪emoji，无点击跳转）

范围外
手机号/第三方登录（仅邮箱）

密码找回（MVP假设用户记住密码）

图片压缩/滤镜

公开社区/评论/排行榜

AI多轮对话式推荐

音乐播放器集成（仅显示歌曲名称+歌手+封面图）

多人协作（超过2人的邀请协助，MVP支持1对1）

后台管理/内容审核

营业时间/电话等信息的用户纠错（依赖高德数据准确性）

年度总结（MVP不包含，后续迭代）

打卡记录列表点击跳转点位详情页（MVP仅展示列表，不支持点击跳转）

个人中心打卡记录列表的图片多图展示（MVP仅展示第一张缩略图）

月度总结自动通知（用户主动点击查看）

店铺详情卡片无共同回忆时的引导文案（直接不显示Tab或Tab为空）

6. 用户流程
流程A：注册并完成第一次探店记录

用户打开Web App → 登录页 → 注册（邮箱+密码+验证码）

登录后进入地图首页

点击“添加点位”按钮 → 弹出搜索面板

输入店铺名称，选填城市 → 选择目标店铺 → 自动填充基础信息

填写个人评价：喜爱值、情绪emoji、上传图片、文字评价

点击“AI推荐标签” → 高亮选中需要的标签

选择“是否邀请好友共同评价” → 选“否”

保存 → 点位出现在地图上

进入个人中心 → 打卡记录列表显示该记录（emoji、喜爱值、图片缩略图、店铺名）

流程B：一起去探店，邀请协助填写

用户A和好友B一起去某家店吃饭

用户A重复流程A的步骤1-6

在“是否邀请好友共同评价”处 → 选择“是” → 选择好友B

保存 → 用户A的评价保存，邀请记录生成

好友B收到通知 → 填写评价 → 提交

双方地图上该店铺显示“共同去过”标记

用户A进入店铺详情卡片 → “共同去过”Tab显示双方评价

用户A进入个人中心 → 共同回忆 → 选择好友B → 看到一起去过的店铺列表

流程C：查看月度总结

用户进入个人中心 → 点击“月度总结”

默认显示当前月份（如2026年4月）

查看统计卡片：点位数量、平均喜爱值、最常标签、情绪分布

查看AI推荐音乐 → 点击“换一首”刷新推荐

滚动查看当月打卡记录列表（店名、日期、喜爱值、情绪emoji）

流程D：查看与特定好友的共同回忆

用户进入个人中心 → 点击“共同回忆”

看到好友列表（仅显示有共同探店记录的好友）

点击某位好友（如好友B）→ 进入详情页

查看总结：共同探店数量、口味相似度、情绪重合度 + 音乐推荐

滚动查看所有一起去过的店铺列表（店名、地址、双方评价摘要）

流程E：语义化搜索

用户在地图页点击“语义搜索”输入框

输入“惊喜的餐厅” → 系统匹配🥰情绪的点位 → 返回结果

8. 非功能性期望
性能: 共同回忆聚合查询 ≤ 500ms；月度总结统计 ≤ 1秒；打卡记录列表加载 ≤ 300ms（每页20条）

可靠性: 同一POI ID的店铺关联准确率 ≥ 99%

可用性: 个人中心导航清晰，2步内找到共同回忆/月度总结

安全性: 密码bcrypt加密；JWT有效期7天

响应速度: 地图加载 ≤ 2秒；AI标签推荐 ≤ 3秒

9. 风险和未决问题
产品风险:

用户不看个人中心 → 在地图首页增加入口提示（如头像/昵称区域）

月度总结用户无感知 → 用户主动点击查看（MVP不做通知）

技术风险:

月度总结统计随着数据量增大变慢 → 增加预计算表（每月定时任务生成）

共同回忆按好友分组查询可能较慢 → 使用数据库索引（userId, collaborationId）

用户体验风险:

打卡记录列表图片加载慢 → 图片懒加载 + 缩略图

月度总结音乐推荐重复 → “换一首”时增加随机种子

依赖风险:

高德API限额 → 初期日调用5000次足够

千问API成本 → 新用户7000万免费Token

Supabase服务稳定性 → 有免费套餐SLA，可接受

模糊假设（已做出的决策）:

假设个人头像使用UI Avatar（昵称首字母/默认图标），MVP不实现自定义上传

假设打卡记录列表仅展示，不支持点击跳转（后续迭代）

假设月度总结按月统计新增点位，不包含历史累计

假设年度总结后续迭代，MVP不做
